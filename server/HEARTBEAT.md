# 自动游戏测试与开发流程

## 核心原则

**有意义 > 有任务**
- 宁缺毋滥，不做无意义的任务
- 队列空时 → 真实体验测试游戏
- 每一次执行都要产生实际价值

---

## 任务执行策略

### 饱和模式 (用量 < 80%)
```
优先级:
1. P0: 修复已知问题 (API报错/功能异常)
2. P1: 完善核心功能 (签到/首充/冲榜)
3. P2: 新功能开发 (宗门/灵兽)
4. P3: 优化迭代 (UI/动画)
```

### 保守模式 (用量 > 80%)
```
只执行: P0 紧急修复
```

### 队列为空时 → 策划体验 + 任务生成模式

当任务队列为空，**立即触发**策划/玩家体验，不退出：

**Phase 1: 策划体验 (Planner Agent)**
- 启动策划角色
- 深度体验游戏现有功能
- 对照梦幻西游/DNF/等经典游戏找差距
- 搜索互联网成功游戏设计经验
- **输出: 新功能设计提案** (写入 `docs/PROPOSALS/{date}.md`)

**Phase 2: 玩家体验 (QA Agent)**
- 以真实玩家视角走查游戏
- 记录操作卡点/体验问题/数值不平衡
- 验证已有功能是否流畅
- **输出: 体验报告** (写入 `docs/EXPERIENCE_REPORT_{timestamp}.md`)

**Phase 3: 任务生成**
- 根据 Phase 1/2 的发现，生成具体开发任务
- 写入 `cron-tasks-todo.txt` 的"待执行"区
- 任务格式: `[优先级] 功能名 - 简短描述`

**Phase 4: 执行或汇报**
- 如果用量充足 → 立即开始执行新任务
- 如果用量不足 → 仅生成任务，等待触发

### 关键原则
- **永远不退出**，队列为空 = 开始创作
- 策划和玩家体验本身就是工作，不是等待
- 每次心跳至少产出: 1份体验报告 + 若干新任务提案

---

## 心跳流程 (优化版)

```
1. 自检 (前置必做)
   ├── 文件路径一致性: 检查cron任务引用的文件是否存在
   ├── Git状态一致性: 检查worktree是否与main同步
   ├── 服务健康检查: 检查前端/后端端口是否响应
   └── API路由检查: 前端调用的API后端是否实现
   └── 发现问题 → 立即自动修复 → 修复失败则告警用户

2. 系统资源检查
   ├── CPU负载 > 80% → 暂停所有任务
   ├── 内存使用 > 85% → 暂停所有任务
   ├── 硬盘使用 > 90% → 暂停所有任务 + 告警
   └── 资源正常 → 继续任务检查

3. 用量检查
   ├── < 40%: 执行多个任务 + 真实体验
   ├── 40-80%: 执行任务
   ├── > 80%: 暂停
   └── 最后10分钟窗口（remains_hours < 0.17）: 触发 auto-art-before-refresh.sh 自动生图（每100次剩余生成1张）

4. 任务队列检查
   ├── 有P0/P1任务 → 执行
   ├── 只有P2/P3 → 评估价值，决定是否执行
   └── 队列空/无意义 → 真实体验模式

5. 真实体验 (当无合适任务时)
   ├── 策划深度体验 → 发现问题
   ├── 玩家视角体验 → 发现问题
   └── 立即修复问题
```

### 自检规则 (自动执行)

**自检项目 | 检测方法 | 自动修复**

| 检查项 | 方法 | 修复 |
|--------|------|------|
| cron引用文件 | grep cron-tasks-todo.txt 路径存在 | 移动文件到正确位置 |
| worktree同步 | git rev-parse HEAD vs origin/main | 自动 reset --hard origin/main |
| 服务健康 | ss -tlnp 检查端口 | 重启服务 |
| API路由 | 前端/api/ vs 后端/routes/ | 创建stub路由文件 |
| 模块路径 | require路径是否存在 | 修复require语句 |

---

## 系统资源检查标准

| 资源 | 正常 | 警告 | 危险 |
|------|------|------|------|
| CPU负载 | < 60% | 60-80% | > 80% |
| 内存使用 | < 70% | 70-85% | > 85% |
| 硬盘使用 | < 75% | 75-90% | > 90% |

**响应动作**：
- 危险: 立即暂停所有任务，通知用户
- 警告: 减少任务并发，延后非紧急任务
- 正常: 正常执行任务

---

## 输出

- `docs/EXPERIENCE_REPORT_{timestamp}.md` - 体验报告
- `docs/ISSUES.md` - 问题清单 (发现即记录)
- 修复任务 (从体验中发现)

## 📋 统一汇报任务 (所有任务执行完毕后)

### 触发条件
当其他任务队列全部执行完毕后，执行此汇报任务。

### 汇报内容（仅限今日）

**1. 今日 MiniMax Token Plan 用量（真实API）**
- 调用 `bash scripts/check-minimax-usage.sh` 查真实用量
- 展示 MiniMax-M2.7 周期用量%（已用/总额），所有模型共用同一配额池
- 展示本周总量已用%（已用/52500）
- 本周期刷新倒计时
- ⚠️ 阈值按 Token Plan 实际用量判断：<50%充足/50-80%正常/80-95%紧张/>95%暂停

**2. 今日任务执行**
- 今日执行的任务列表（含结果）
- 今日新发现/修复的问题
- 今日完成的功能

**3. 当前进行中**
- 正在进行的功能
- 队列中待处理任务

**4. 系统状态**
- 游戏前端端口状态
- 后端服务状态
- 系统资源使用情况

**5. 紧急问题（如有）**
- P0/P1 紧急问题
- 需要人工介入的事项

### 汇报对象
统一向用户（飞书）发送汇报消息，格式清晰，使用表格和列表。

### 输出文件
同时生成 `memory/reports/{date}-report.md` 备份汇报内容。
